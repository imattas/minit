# v1.0.0 Readiness Plan

## Goal

Ship `minit` v1.0.0 only after the repository has repeatable evidence for the first cautious daily-driver-candidate claim.

## Scope

This plan keeps v1.0.0 conservative. It does not make `minit` a broad systemd replacement, add a journal, device manager, network manager, or user session manager. It requires that release claims match verified boot, service, shutdown, packaging, distro-rootfs, and security evidence.

## Phase 1: v1 Gate and Roadmap

- Add a strict `tools\verify-v1-readiness.ps1` wrapper.
- Require kernel and BusyBox inputs for full v1 validation.
- Run existing release verification with extended VM stress.
- Run the security gate.
- Run Alpine distro-rootfs validation by default.
- Support required Debian and Arch distro-rootfs validation when local rootfs inputs exist.
- Emit machine-readable readiness evidence.
- Document the gate and link it from the README.
- Make VM transcript capture deterministic enough for long local release runs.
- Retry VM smoke steps with a bounded attempt count while keeping source, build, package, and security steps single-attempt.

Acceptance:

- The new script parses under PowerShell.
- Missing full-VM inputs fail closed.
- Source-only preflight runs the source release and security gates.
- Full VM preflight runs release, security, extended boot-loop stress, and Alpine distro-rootfs gates.
- Docs describe what source-only mode does not prove.

## Phase 2: Package Install and Rollback Gate

- Add a disposable rootfs install script that copies packaged artifacts into a root tree.
- Add an uninstall or rollback script for the same root tree.
- Add tests that verify install layout, ownership-sensitive paths, and rollback cleanup.
- Extend VM validation to boot the installed layout rather than only a hand-built initramfs profile.

Acceptance:

- Install and rollback are both idempotent in a disposable rootfs.
- A VM boot can use the installed rootfs layout.
- Failure leaves a clear recovery path.

## Phase 3: Full Disk VM Gate

- Build a disposable disk image from the installed rootfs.
- Boot the image through QEMU with `minitd` as PID 1.
- Verify `minitctl status`, `list`, service restart, logs/events, shutdown, and reboot.
- Preserve serial logs as evidence artifacts.

Acceptance:

- The gate proves a disk boot, not only initramfs boot.
- Shutdown does not hang with known stuck-process fixtures.
- Logs identify the kernel, rootfs profile, and minit build.

## Phase 4: Distro Profile Hardening

- Keep PID 1 responsibilities narrow and document external manager boundaries.
- Provide examples for getty, sshd, networking handoff, dbus, and display/login target handoff.
- Mark unsupported distro/service features explicitly.
- Add acceptance smokes for every shipped profile.

Acceptance:

- Unsupported features fail closed or are explicitly ignored by documented conversion tools.
- Shipped profiles boot in disposable VM validation.

## Phase 5: Security Hardening Expansion

- Expand seccomp and no-new-privileges proof coverage.
- Verify UID/GID switching and environment allowlist behavior in VM.
- Confirm control socket permissions and unsafe unit-name rejection.
- Keep fuzz targets in the security gate, with platform-specific skip rules documented.

Acceptance:

- Security options either work or reject the unit before execution.
- VM proof covers at least one denied syscall, privilege boundary, and unsafe config rejection.

## Phase 6: Observability and Incident Evidence

- Preserve boot/shutdown timeline output in v1 evidence.
- Preserve recent lifecycle events and logs from VM smokes.
- Document persistent logging handoff because PID 1 does not replace a journal.

Acceptance:

- Failed v1 gates leave enough serial or command output to debug the failing phase.
- Release notes do not imply persistent journaling.

## Phase 7: Release Candidate and v1.0.0 Tag

- Bump crate/package versions to `1.0.0`.
- Add `docs\releases\v1.0.0.md` with exact verified scope.
- Run the full v1 readiness gate.
- Commit, tag `v1.0.0`, push, and create a non-prerelease GitHub release.
- Confirm GitHub Actions pass for `master` and `v1.0.0`.

Acceptance:

- The release is not tagged until fresh full-gate evidence exists.
- The release notes list remaining limits plainly.
